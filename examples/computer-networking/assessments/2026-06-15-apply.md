## Task

You type `https://shop.example.com` into a browser on a fresh machine (empty
caches). Walk through, in order, what happens at the network level up to the
moment the first byte of the page is requested. Name the protocol at each step
and say *why* it's used there. Then: one of these steps uses UDP — which, and
what would break if it used TCP instead?

## Your explanation

1. **DNS resolution** — the browser needs an IP for `shop.example.com`. It asks
   the OS resolver, which walks browser → OS → recursive resolver → root → TLD
   (`.com`) → authoritative server for `example.com`. Uses **UDP** (port 53):
   queries are tiny and a lost one is just retried, so the handshake overhead of
   TCP isn't worth it.
2. **TCP three-way handshake** — now that we have the IP, open a connection:
   SYN → SYN-ACK → ACK. **TCP** because the page must arrive complete and in
   order.
3. **TLS handshake** — it's `https`, so negotiate encryption over that TCP
   connection.
4. **HTTP request** — finally `GET /` is sent; first byte requested.

UDP step = DNS. If DNS used TCP, every lookup would pay a full handshake
round-trip first — much slower, for a request that's usually one small packet.
(Caveat: DNS does fall back to TCP for large responses, e.g. DNSSEC.)
